The Problem

  • AI agents are powerful but hard to operationalize
  • Running an agent locally ≠ running it in CI
  • No standard for security boundaries, tool manifests, or output contracts
  • Multi-agent pipelines require custom glue code every time

One Definition, Every Environment

Workflow YAML

agent:
  runtime: claude-code
  model: claude-sonnet-4-20250514
  system_prompt: prompts/review.md

tools:
  capabilities:
    - repository:read

safety:
  trust: sandboxed
  resources:
    timeout: 300

What It Declares

  • Which agent runs the work
  • What tools the agent can access
  • What constraints are enforced
  • What inputs are required
  • What output schema is expected

Same file → agentry run locally
Same file → agentry ci generate for CI

Four-Layer Architecture

Agentry — orchestration, CLI, composition engine
Runner — execution environment (Docker sandbox, in-process)
Agent — coding agent runtime (Claude Code, extensible)
Model — LLM (Claude, with agent handling all API interaction)

Agentry never makes direct LLM API calls.
Runners own the environment. Agents own the intelligence.

Security by Default

Least Privilege

Tool manifest enforcement. Agents only access declared capabilities. repository:read means no writes.

Sandbox Isolation

Docker containers with CPU/memory limits, read-only mounts, non-root user, DNS-based egress filtering.

Signing & Audit

Ed25519 signatures over safety blocks. Security audit diffs between workflow versions. Preflight checks before execution.

Multi-Agent Composition

Triage → Decompose → Summarize

DAG Scheduling

  • Independent nodes run concurrently
  • Data flows between nodes via output references
  • Single-node debugging with --node

Failure Policies

  • abort — halt the pipeline
  • retry — re-execute with backoff
  • skip — pass failure downstream

CI Generation

$ agentry ci generate --target github workflows/code-review.yaml

What It Produces

  • Ready-to-commit GitHub Actions YAML
  • Minimal token permissions from tool manifest
  • Configurable triggers (PR, push, schedule, issues)
  • Auto-detection of CI environment at runtime

The Vision: Self-Hosting

Agentry developing itself.

Research → Spec → Plan → Dispatch → Implement → Validate 
$ agentry run workflows/develop.yaml \
    --input spec=docs/specs/07-spec-feature.md

Full development lifecycle as a composed workflow. No external plugins.

Designed to Extend

Add an Agent Runtime

Implement AgentProtocol, register in AgentRegistry. One protocol, one registration.

class MyAgent:
  def execute(self,
    task: AgentTask
  ) -> AgentResult: ...

  def check_available(self)
    -> AgentStatus: ...

Add a CI Target

Implement EnvironmentBinder. Entry-point based discovery.

# pyproject.toml
[project.entry-points.
  "agentry.binders"]
gitlab =
  "my_pkg:GitLabBinder"
