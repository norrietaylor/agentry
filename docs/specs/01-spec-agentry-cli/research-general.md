# Research Report: Agentry — Portable Agentic Workflow Orchestration

> Generated by cw-research on 2026-03-20
> Codebase: agentry (greenfield — PRD only)

## Summary

- **Greenfield project** with a comprehensive PRD (`agentry-prd.md`) and no existing code. All findings are ecosystem research and architectural validation against the PRD's technical decisions.
- **Python is the correct choice.** The AI agent ecosystem (LangGraph, CrewAI, Claude Agent SDK, Anthropic/OpenAI SDKs) is overwhelmingly Python-first. All PRD-referenced libraries (docker-py 7.1.0, pyyaml 6.0.3, jsonschema 4.26.0) are production-ready and actively maintained.
- **Agentry fills a genuine market gap.** No existing tool combines declarative YAML workflow definitions + cross-environment portability (local <-> CI) + structural security enforcement (sandbox + output validation + side-effect allowlists). CrewAI has YAML but no security; LangGraph has graphs but requires code; Microsoft Agent Framework has security but is Azure-centric.
- **The security model is the differentiator.** Prompt injection is ranked LLM01 by OWASP with 50-84% attack success rates. Structural constraints (typed schemas, allowlists, sandbox enforcement) are more robust than prompt-level defenses. The PRD's three-layer output validation pipeline and mandatory setup phase with preflight checks are well-aligned with Anthropic and OpenAI safety frameworks.
- **Key implementation risks:** Docker network egress filtering requires iptables manipulation (complex, Linux-only for full isolation). The environment binder is the hardest engineering component — the PRD correctly estimates 2,000-5,000 lines per binder. GitHub Actions fork PR restrictions (read-only tokens, no secrets) are a significant constraint that the binder must handle explicitly.

## Architecture & Patterns

### Design Patterns

**Four-Layer Architecture** (from PRD):
1. **Definition** — Workflow spec (YAML, environment-agnostic)
2. **Safety** — Security Envelope (enforced before binding)
3. **Resolution** — Environment Binder (abstract -> concrete)
4. **Execution** — Runtime Engine (lifecycle management)

This maps to the **compiler backend metaphor**: workflow spec = source code, execution plan = binary, environment binder = compiler backend. LLVM's target-independent IR provides the canonical reference for this pattern.

**Plugin Architecture**: Python Protocols (structural subtyping, PEP-544) preferred over ABCs for binder interfaces. Allows third-party binders without requiring inheritance from Agentry base classes.

**DAG Composition**: `graphlib.TopologicalSorter` (stdlib) for dependency resolution + `asyncio.TaskGroup` (3.11+) for parallel execution of independent nodes. Per-node timeout via `asyncio.timeout()`.

**Dependency Injection**: Environment binder selected by runtime detection (`GITHUB_ACTIONS` env var) or explicit CLI flag. Concrete executor injected into workflow engine — no service locator anti-pattern.

### Module Boundaries

| Module | Responsibility | Dependencies |
|--------|---------------|-------------|
| `cli/` | User-facing commands, output formatting | `core/`, `binders/` |
| `core/` | Workflow parsing, DAG execution, validation | `sandbox/`, `llm/`, `security/` |
| `sandbox/` | Container lifecycle, network isolation | docker-py, subprocess |
| `binders/` | Abstract->concrete resolution per environment | GitHub API, git CLI |
| `security/` | Envelope enforcement, signing, preflight | cryptography, jsonschema |
| `llm/` | Provider-abstracted model client | Anthropic SDK, OpenAI SDK |

### Key Abstractions

**Binder Protocol** (most critical interface):
```python
class EnvironmentBinder(Protocol):
    def resolve_inputs(self, workflow: WorkflowDef, context: EnvContext) -> ResolvedInputs: ...
    def bind_tools(self, tool_manifest: list[ToolDecl], context: EnvContext) -> list[BoundTool]: ...
    def map_outputs(self, output_contract: OutputContract, context: EnvContext) -> OutputMapping: ...
    def generate_pipeline_config(self, workflow: WorkflowDef) -> str: ...
```

**OutputValidator** (three-layer pipeline):
1. JSON Schema validation (`jsonschema.validate()`)
2. Side-effect allowlist check (declared vs attempted actions)
3. Output path enforcement (`pathlib.Path.is_relative_to()`)

### Conventions

- **Pydantic v2 models** for all configuration with discriminated unions for polymorphic types
- **Strict YAML parsing**: Unknown keys are errors, not warnings
- **Fail closed**: Ambiguous security situations halt execution
- **Click error hierarchy**: `ClickException` for user-facing, `BadParameter` for invalid options

### Deep-Dive Findings

**asyncio DAG Execution Pattern:**
```
While topological_sorter has ready nodes:
  For each ready node:
    create_task(execute_agent(node_id, inputs)) in TaskGroup
  On TaskGroup exit, collect results
  Mark nodes done → next batch
```

Results stored in shared dict keyed by node ID. Downstream nodes read dependencies from dict. Per-node timeout via `asyncio.timeout(seconds)`.

**Pydantic YAML Loading:**
1. `yaml.safe_load()` → dict
2. `WorkflowModel.model_validate(dict)` → typed model
3. Custom validators via `@field_validator(mode='before')` for preprocessing
4. `@model_validator(mode='after')` for cross-field validation
5. `Optional[CompositionModel] = None` for conditional blocks

### Integration Points

**GitHub Actions Binder:**
- Event payload parsing (`pull_request`, `push`, `workflow_dispatch`)
- GITHUB_TOKEN scope verification via `GET /user` → `X-OAuth-Scopes` header
- PR comments via Issues API (`POST /repos/{owner}/{repo}/issues/{number}/comments`)
- Check annotations via Checks API (`POST /repos/{owner}/{repo}/check-runs`)
- Artifact uploads via `actions/upload-artifact@v4`

**Critical constraint — Fork PRs:**
- `pull_request` from forks: read-only GITHUB_TOKEN, no secrets access
- Workaround: `pull_request_target` event (runs against base repo code with write access)
- Agentry binder must detect fork context and fail fast with actionable diagnostic

**Git Worktrees:**
- `git worktree add <path> [branch]` for parallel agent filesystem isolation
- Sub-second creation, shared .git history
- Each agent in a composition gets its own worktree
- Cleanup via `git worktree remove` after execution

**LLM Provider Abstraction:**
- Workflow definition specifies provider + model identifier
- Client resolves to appropriate SDK (Anthropic, OpenAI)
- API keys from environment variables, never in workflow definitions
- Retry policy (exponential backoff) declared in workflow definition

### Internal Data Flow

```
Workflow YAML → Parser (Pydantic) → WorkflowDef
    → Security Envelope (strip excess capabilities)
    → Environment Binder (resolve inputs, bind tools, map outputs)
    → Setup Phase (preflight checks, sandbox provision)
    → Runtime Engine (asyncio DAG executor)
    → Output Validator (schema → allowlist → paths)
    → Emission (PR comment, terminal output, artifact)
```

## Data Models & API Surface

### Data Models (from PRD Section 6)

| Entity | Storage | Mutability | Key Fields |
|--------|---------|------------|-----------|
| Workflow Definition | YAML files in repo | Versioned via git | identity, inputs, tools, model, safety, output, composition |
| Environment Binding | In-memory, ephemeral | Immutable once created | resolved inputs, tool endpoints, credential refs |
| Security Envelope | In-memory + execution record | Immutable | sandbox config, schema validator, allowlist, permitted paths |
| Setup Manifest | JSON alongside execution record | Immutable | container image, mount paths, tool versions, credential fingerprints |
| Execution Record | JSON files (local) / CI artifacts | Append-only | definition version, binding, I/O, tool invocations, timing, status |
| Composition Graph | Embedded in Workflow Def | Versioned via git | DAG nodes, data-flow edges, failure policies |
| Registry Entry | Local index file | Regenerated on scan | name, description, I/O types, environments |

### CLI Surface (5 core commands)

| Command | Purpose | Key Options |
|---------|---------|------------|
| `agentry run` | Execute a workflow | `--input`, `--target`, `--verbose` |
| `agentry validate` | Check definition correctness | `--security-audit` |
| `agentry setup` | Prepare environment without running | Produces setup manifest |
| `agentry ci generate` | Produce CI pipeline config | `--target github` |
| `agentry registry list` | Show available workflows | (none) |

### Workflow Definition Schema (key blocks)

1. **Identity Block**: name, version, description
2. **Input Contract**: typed abstract inputs (git-diff, repository-ref, document-ref)
3. **Tool Manifest**: capability-based declarations (repository:read, pr:comment)
4. **Model Configuration**: provider, identifier, temperature, max_tokens, system_prompt, retry
5. **Safety Block**: trust level, resources (CPU/memory/timeout), filesystem paths, network allowlist
6. **Output Contract**: JSON Schema, side-effect allowlist, output paths, budget
7. **Composition Block** (optional): DAG of workflow references with failure policies

### Output Validation Pipeline

Three mandatory layers, executed in order:

1. **Schema Validation**: JSON Schema check on agent output structure
2. **Side-Effect Allowlist**: Compare attempted actions against declared permissions
3. **Output Path Enforcement**: Verify writes target only declared destinations

Failure at any layer halts emission. Structured error returned indicating which layer failed and why.

## Competitive Landscape

### Direct Comparisons

| Feature | Agentry | CrewAI | LangGraph | MS Agent Framework | Claude SDK |
|---------|---------|--------|-----------|-------------------|------------|
| Definition format | Declarative YAML | YAML (agents + tasks) | Python code | YAML/JSON + code | Python code |
| Environment portability | Local + any CI | Local only | Python anywhere | Azure-centric | Local only |
| Security envelope | Structural enforcement | None | None | Governance toolkit | Sandbox guidance |
| Output validation | 3-layer mandatory | Developer responsibility | Developer responsibility | Basic schema | None |
| Side-effect allowlists | Built-in | None | None | Policy-based | None |
| Sandbox execution | Docker + graduated tiers | None | None | Process-level | Recommended, not enforced |
| Composition model | DAG with failure policies | Linear task chain | Stateful graph | Graph-based | Subagents |
| Setup/preflight | Mandatory gate | None | None | Similar concept | None |

### Agentry's Unique Value Proposition

**The "Single Artifact Guarantee":** One YAML file that is simultaneously portable, secure, auditable, maintainable, composable, and validated. No other tool combines all six properties.

**Market gaps Agentry fills:**
1. **Declarative + Portable + Secure**: No tool combines all three today
2. **Validation Stack**: Schema + allowlists + path constraints + output budgets, enforced by runtime
3. **Environment Binding as first-class abstraction**: Adding a CI system = implementing one binder interface
4. **Composition failure semantics**: Per-node abort/skip/retry with structured failure objects
5. **Mandatory setup phase**: Preflight checks prevent 20-minute runs failing on last API call

## Implementation Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Docker network egress filtering requires iptables (Linux-specific) | High | Graduated sandbox strategy; DNS-based filtering as lighter alternative |
| Environment binder complexity (2-5K LOC per binder) | High | Start with Local binder only; GitHub Actions as second milestone |
| GitHub fork PR read-only tokens | Medium | Detect fork context in binder; use `pull_request_target` with base-repo-only execution |
| YAML schema evolution and versioning | Medium | Explicit `version` field; strict validation with unknown-key rejection |
| Container image management adoption friction | Medium | Zero-config base image; dependency block for incremental complexity |
| Prompt injection through untrusted inputs (diffs, issues) | High | Three-layer output validation; typed contracts between agents; sandbox capability limits |
| Workflow definition tampering via malicious PRs | Medium | Ed25519 signing of safety block; `validate --security-audit` for change detection |
