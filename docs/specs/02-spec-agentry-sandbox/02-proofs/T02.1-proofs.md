# T02.1 Proof Summary: Isolated Docker Network Creation and Teardown

## Task

Implement `NetworkManager` class in `src/agentry/runners/network.py` that creates
isolated Docker networks per sandbox execution using docker-py. Network uses bridge
driver with `internal=True` (no default internet connectivity). Provides
`create_network(execution_id) -> network_id` and `teardown_network(network_id)`.

## Implementation

**Files created:**
- `src/agentry/runners/__init__.py` — Package init for runners module
- `src/agentry/runners/network.py` — `NetworkManager` implementation
- `tests/unit/test_network_manager.py` — Unit tests with mocked docker client

**Key design decisions:**
- docker-py imported lazily with `_DOCKER_AVAILABLE` flag; `docker_client` parameter
  allows injection in tests without requiring a Docker daemon
- `internal=True` on network creation disables the default route, enforcing
  no outbound internet connectivity from containers on the network
- Labels (`agentry.execution_id`, `agentry.managed`) enable discovery and cleanup
- Teardown is idempotent: 404/"not found" responses are silently ignored to support
  `finally` block usage
- `NetworkCreationError` and `NetworkTeardownError` are distinct exception types
  for precise error handling in callers

## Test Results

20/20 unit tests pass with mocked docker client. Tests cover:
- `create_network()`: returns network ID, uses bridge driver, sets `internal=True`,
  constructs network name from prefix + execution_id, sets labels, raises
  `NetworkCreationError` on API failure
- `teardown_network()`: calls `networks.get()` then `network.remove()`, is idempotent
  on 404/not-found errors, raises `NetworkTeardownError` on unexpected failures
- Constructor: raises `RuntimeError` when docker-py absent, accepts explicit client,
  uses default prefix `agentry-net`
